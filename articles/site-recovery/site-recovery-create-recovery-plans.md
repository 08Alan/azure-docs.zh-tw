# <a name="create-a-recovery-plan-by-using-site-recovery"></a>使用 Site Recovery 建立復原方案

本文說明如何在 [Azure Site Recovery](site-recovery-overview.md) 中建立和自訂復原方案。

建立復原方案以執行下列作業：

* 定義會一起容錯移轉而後一起啟動的機器群組。
* 將機器分到同一個復原方案群組，藉以建立機器間的相依性。 例如，若要執行容錯移轉並帶出特定的應用程式，請將該應用程式的所有 VM 納入相同的復原方案群組。
* 執行容錯移轉。 您可以透過復原方案執行測試，以及執行規劃或未規劃的容錯移轉。

## <a name="why-use-a-recovery-plan"></a>為何使用復原方案

復原方案可藉由建立您可管理的小型獨立單位，協助您針對系統性復原程序做準備。 這些單位通常代表您環境中的應用程式。 復原方案可協助您定義 VM 啟動的順序。 您也可以使用復原方案，在復原期間自動執行一般工作。

> [!TIP]
> 其中一個可確認您已為雲端移轉或災害復原做好準備的方法，就是確保將每個應用程式納入復原方案中。 此外，確保每個復原方案都已通過復原至 Azure 的測試。 有了這樣的準備狀態，您就能信心十足地將整個資料中心移轉或容錯移轉至 Azure。
 
復原方案有三個重要價值主張：

* 建立應用程式模型以擷取相依性。
* 將大多數復原工作自動化以縮短 RTO。
* 測試容錯移轉以做好災害因應準備。

### <a name="model-an-application-to-capture-dependencies"></a>建立應用程式模型以擷取相依性

復原方案是一組通常構成一個應用程式且一起容錯移轉的虛擬機器。 使用復原方案建構，您即可增強復原方案群組來擷取應用程式特定屬性。 

在本文中，我們會使用可能具有 SQL 後端、中介軟體和 Web 前端的典型三層式應用程式。 您可以自訂復原方案，協助確保虛擬機器在容錯移轉後以正確的順序啟動。 SQL 後端應最先啟動，中介軟體應接著啟動，而 Web 前端應最後啟動。 此順序可確保應用程式在最後一部虛擬機器啟動時已在運作。 

例如，當中介軟體啟動時，它會嘗試連線到 SQL 層。 復原方案可確保 SQL 層已在執行中。 讓前端伺服器最後啟動也有助於確保在所有元件都已啟動並執行，且應用程式已準備好接受要求之前，使用者不會連到應用程式 URL。 若要建置這些相依性，請自訂復原方案來新增群組，然後選取虛擬機器。 若要在群組之間移動虛擬機器，請變更虛擬機器的群組。

![Site Recovery 中範例復原方案的螢幕擷取畫面](./media/site-recovery-create-recovery-plans/rp.png)

完成自訂之後，您可以將確切的復原步驟以視覺化方式呈現。 以下是在復原方案的容錯移轉期間所執行的步驟順序：

1. 關閉步驟會嘗試關閉內部部署虛擬機器。 (測試容錯移轉除外，其間主要網站需要繼續執行。)
2. 關閉嘗試會以平行方式觸發復原方案中所有虛擬機器的容錯移轉。 容錯移轉步驟會使用複寫的資料準備虛擬機器磁碟。
3. 啟動群組會以其順序執行，並啟動每個群組中的虛擬機器。 首先執行群組 1，接著執行群組 2，最後執行群組 3。 如果任何群組中有多部虛擬機器 (例如負載平衡的 Web 前端)，所有虛擬機器都會以平行方式啟動。

> [!TIP]
> 跨群組排序可確保遵守各種應用程式層之間的相依性。 而適用的平行處理能力可改善應用程式復原的 RTO。

   > [!NOTE]
   > 單一群組中的機器會以平行方式容錯移轉。 不同群組中的機器會依群組順序容錯移轉。 只有群組 1 的所有機器都已容錯移轉並啟動之後，群組 2 的機器才會開始其容錯移轉。

### <a name="automate-most-recovery-tasks-to-reduce-rto"></a>將大多數復原工作自動化以縮短 RTO

復原大型應用程式可能是相當複雜的工作。 在混亂的情況下要記得太多手動步驟，不僅困難，也容易讓程序出錯。 而且，有可能是其他未注意到應用程式複雜性，但卻實際觸發容錯移轉的人。 

您可以使用復原方案來自動執行您需要在每個步驟採取的必要動作。 您可以使用 Azure 自動化 Runbook 來設定必要的動作。 使用 Runbook 時，您可以將一般復原工作自動化，如以下範例中的工作。 對於無法自動化的工作，您可以將手動動作插入復原方案中。

* **容錯移轉後 Azure 虛擬機器上的工作**：這些通常是能夠連線至虛擬機器的必要工作。 範例：
    * 在容錯移轉後於虛擬機器上建立公用 IP 位址。
    * 將網路安全性群組指派給容錯移轉虛擬機器上的 NIC。
    * 將負載平衡器新增至可用性設定組。
* **容錯移轉後虛擬機器內的工作**：這些工作會重新設定應用程式，以便讓應用程式在新環境中繼續正常運作。 範例：
    * 修改虛擬機器內的資料庫連接字串。
    * 變更 Web 伺服器組態或規則。

> [!TIP]
> 建立完整的復原方案，以使用自動化 Runbook 將復原後工作自動化，即可達成單鍵容錯移轉並將 RTO 最佳化。

### <a name="test-failover-to-be-ready-for-a-disaster"></a>測試容錯移轉以做好災害因應準備

您可以使用復原方案來觸發容錯移轉或測試容錯移轉。 在進行容錯移轉之前，一律先在應用程式上完成測試容錯移轉。 測試容錯移轉可協助您檢查應用程式是否會在復原網站上啟動。 如果您在設定時遺漏了某個項目，您可以輕鬆地觸發清除，然後重做測試容錯移轉。 請執行測試容錯移轉多次，直到您確定應用程式可順暢復原為止。

![Site Recovery 方案中範例復原方案的螢幕擷取畫面](./media/site-recovery-create-recovery-plans/rptest.png)

> [!TIP]
> 因為每個應用程式都是唯一的，所以您必須建置針對每個應用程式自訂的復原方案。 
>
> 此外，在現今以資料中心為主的動態環境中，應用程式及其相依性都會頻繁改變。 若要確保復原方案是最新的，請每季為您的應用程式進行一次測試容錯移轉。

## <a name="create-a-recovery-plan"></a>建立復原計畫

1. 選取 [復原方案] > [建立復原方案]。
   指定復原方案的名稱，以及來源和目標伺服器。 來源位置必須有已啟用容錯移轉和復原功能的虛擬機器。 請根據要參與復原方案的虛擬機器，選擇來源和目標。 

   |案例                   |來源               |目標           |
   |---------------------------|---------------------|-----------------|
   |Azure 至 Azure             |Azure 區域         |Azure 區域     |
   |VMware 至 Azure            |組態伺服器 |Azure            |
   |Virtual Machine Manager (VMM) 至 Azure               |VMM 顯示名稱    |Azure            |
   |從 Hyper-V 站台到 Azure      |Hyper-V 網站名稱    |Azure            |
   |實體機器至 Azure |組態伺服器 |Azure            |
   |VMM 至 VMM                 |VMM 易記名稱    |VMM 顯示名稱|

   > [!NOTE]
   > 復原方案可以包含具有相同來源和目標的虛擬機器。 VMware 和 System Center Virtual Machine Manager (VMM) 虛擬機器不能在相同的復原方案中。 不過，您可以將 VMware 虛擬機器和實體機器新增至相同的復原方案。 在此情況下，這兩部電腦的來源都是組態伺服器。

2. 在 [選取虛擬機器] 之下，選取您要在復原方案中新增至預設群組 (群組 1) 的虛擬機器 (或複寫群組)。 您可選取的虛擬機器僅限於在來源上受到保護 (如復原方案中所選取) 且對目標進行保護 (如復原方案中所選取) 的虛擬機器。

## <a name="customize-and-extend-recovery-plans"></a>自訂和擴充復原計畫

若要自訂和擴充復原方案，請移至 Site Recovery 復原方案資源窗格。 選取 [自訂] 索引標籤。您可以使用下列選項來自訂和擴充復原方案：

- **新增群組**：您可以將最多 7 個額外復原方案群組新增至預設群組。 然後，您可以將更多機器或複寫群組新增至這些復原方案群組。 群組會依您加入的順序加以編號。 您只能在一個復原方案群組中包含一部虛擬機器或一個複寫群組。
- **新增手動動作**：您可以新增手動動作，在建立復原方案群組之前或之後執行。 執行復原計畫時，它會在您插入手動動作之處停止。 對話方塊會提示您指定手動動作已完成。
- **新增指令碼**：您可以新增指令碼，在建立復原方案群組之前或之後執行。 新增指令碼後，群組中會加入新的動作集。 例如，系統會為群組 1 建立一組前置步驟，並且命名為「群組 1：前置步驟」。 所有前置步驟都會列在此動作集內。 只有在部署 VMM 伺服器後，您才能在主要網站上新增指令碼。 如需詳細資訊，請參閱[將 VMM 指令碼新增至復原方案](site-recovery-how-to-add-vmmscript.md)。
- **新增 Azure Runbook**：您可以使用 Azure Runbook 來擴充復原方案。 例如，您可以使用 Runbook 來自動執行工作，或建立單步驟復原。 如需詳細資訊，請參閱[將 Azure 自動化 Runbook 新增至復原方案](site-recovery-runbook-automation.md)。


## <a name="add-a-script-runbook-or-manual-action-to-a-plan"></a>將指令碼、Runbook 或手動動作新增至方案

將虛擬機器或複寫群組新增至預設復原方案群組之後，您可以將指令碼或手動動作新增至此復原方案群組。

1. 開啟復原計畫。
2. 在 [步驟] 清單中，選取一個項目。 然後，選取 [指令碼] 或 [手動動作]。
3. 指定您要在選取的項目之前或之後新增指令碼或動作。 若要將指令碼的位置向上或向下移動，請選取 [上移] 或 [下移] 按鈕。
4. 如果您新增 VMM 指令碼，請選取 [容錯移轉至 VMM 指令碼]。 在 [指令碼路徑] 中，輸入要共用的相對路徑。 例如，**\RPScripts\RPScript.PS1**。
5. 如果新增自動化 Runbook，請指定 Runbook 所在的 Azure 自動化帳戶。 然後，選取 Azure Runbook 指令碼。
6. 若要確保指令碼會如預期般運作，請執行復原方案的容錯移轉。

指令碼或 Runbook 選項僅適用於下列案例，以及容錯移轉或容錯回復期間。 手動動作同時適用於容錯移轉和容錯回復。


|案例               |容錯移轉 |容錯回復 |
|-----------------------|---------|---------|
|Azure 至 Azure         |Runbook |Runbook  |
|VMware 至 Azure        |Runbook |NA       | 
|VMM 至 Azure           |Runbook |指令碼   |
|從 Hyper-V 站台到 Azure  |Runbook |NA       |
|VMM 至 VMM             |指令碼   |指令碼   |
